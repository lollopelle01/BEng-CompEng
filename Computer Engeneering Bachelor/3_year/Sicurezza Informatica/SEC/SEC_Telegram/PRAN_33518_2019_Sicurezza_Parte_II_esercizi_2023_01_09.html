Sicurezza - Parte II - esercizi - 2023-01-09

Svolgere i tre esercizi descritti di seguito, caricando i file richiesti via EOL esattamente coi nomi specificati.

possibile ricaricare ogni file un numero illimitato di volte (ogni volta verr proposto di sovrascrivere la versione precedente) entro il tempo limite assegnato.


Esercizio 1 - Iptables


Si consideri la rete in figura.



Quali regole vanno inserite su "R" perch
  • Qualsiasi host lato "Client" possa leggere email via IMAP sicuro (TCP/993) da quasiasi host di "internet"
  • Qualsiasi host lato "Client" possa connettersi a "Server" sulle porte TCP 25 e 587
  • Sia i "Client" che il "Server" possano usare "R" come server LDAP (TCP/389)
  • il solo "Client" 10.1.1.1 possa connettersi a "R" via SSH
  • Da "internet" chiunque possa connettersi alla porta TCP/443 del "Server" - si noti che quest'ultimo su una rete privata.
  • Nessun altro tipo di traffico sia ammesso


Esercizio 2 - Buffer overflow

Unzippare il file con l'eseguibile compilato

$ unzip bof.zip
L'eseguibile es ha bisogno dei permessi di esecuzione.

$ cd bof
$ chmod +x ./es
Compito dello studente sfruttare il buffer overflow e ed eseguire lo shellcode dato (una shell).

La tipologia del buffer overflow una di quelle viste a lezione.

Ricordarsi che a lezione abbiamo visto come sia necessario aver disabilitato l'ASLR per far si che l'esercizio riesca.

Hint:
Da root lanciare "echo 0 > /proc/sys/kernel/randomize_va_space"

Lo studente dovr quindi consegnare 3 file

  • Un file   bof.txt  contenente:
    • Il payload finale con il quale viene lanciato l'eseguibile
    • Una descrizione il pi possibile dettagliata dei passaggi eseguiti per ricavare l'overflow che permette di sovrascrivere l'indirizzo di ritorno. Questa descrizione pu includere tutti i tentativi che sono stati fatti oppure soltanto il meccanismo che stato eseguito.
  • Uno screenshot  over.jpg  che mostra il payload che esegue l'overflow che dimostra che siate in grado di controllare l'indirizzo di ritorno e.g. AAAA..AA+BBBB
  • Uno screenshot  exec.jpg  che mostra il payload eseguito su es e la relativa nuova shell ottenuta.
Il livello di dettaglio e precisione della descrizione sar tenuto in conto nella valutazione della prova.


Esercizio 3 -Privilege escalation

Leggete tutto prima di fare qualsiasi cosa!


Scaricare il file change1 e renderlo eseguibile

$ chmod +x ./change1
Il comando apporta una modifica a un file dentro /usr/bin

Fase 1:
  1. ideare un modo di identificare il file modificato e il tipo di modifica apportata.
  2. lanciare sudo ./change1
  3. attuare la strategia ideata al punto 1 per identificare il file modificato e il tipo di modifica apportata.
  4. documentare tutti i passi svolti in modo dettagliato nel file integrity.txt
Fase 2:
catturate i comandi seguenti e l'output in uno screenshot privesc.png
  1. usate come utente sec il file modificato dal comando change1, per inserire nei file in /etc/passwd ed /etc/shadow le righe opportune per "creare" un utente di nome toor con privilegi di root e senza password (ricordate che esistono le man page)
  2. da sec diventate toor e lanciate id
